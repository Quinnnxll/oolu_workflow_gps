# Workflow-GPS v0.6.0 — the Nodeplace release

Workflow-GPS grows from a planning engine into a **self-improving workflow
economy with a face**: a marketplace where contributed workflow nodes are
discovered, assembled into plans, priced by verified history, executed with
human-controlled governance, and paid — plus a desktop shell (and native
single-file app) that puts all of it in a browser tab. **Every run makes the
next plan better**, and money only ever moves on platform-verified success.

## Get it

- **Native app** — download `WorkflowGPS-Shell` for your platform from the
  release artifacts (built by the `build-installers` workflow), double-click,
  and the shell opens in your browser. Data lives in `~/.workflow-gps`.
- **From the ZIP** — unzip, then double-click `setup.bat` (Windows) or run
  `./setup.sh` (macOS/Linux). One private `.venv` inside the folder; nothing
  installed outside it.
- **Developers** — `pip install -e ".[serve]"` then `wfgps desktop --open`.

## Highlights

### A marketplace that plans
- **NodeContract** unifies every node — typed `consumes`/`produces` slots,
  three body kinds (actions / cached script / subgraph), fallbacks, verified
  stats — and compiles to the partial-order DAG the scheduler executes.
  Ordering falls out of slot flow; unrelated steps run in parallel.
- **Goal-based assembly** (`POST /v1/market/assemble`): name the slots you
  want, and the assembler backward-chains through the marketplace's slot
  vocabularies — picking producers by *platform-verified history*, never
  self-declared quality — and returns a runnable contract with per-step
  cleared-price previews and lineage-aware payout forecasts. Missing slots
  report honestly, or become cached script gap-nodes on request.
- **Live economics**: a persisted `PriceBook` clears asks through cost floor →
  competition → value anchor → per-class damping; browsing and previews never
  move the market — only real runs commit prices.

### Every run teaches the planner
- Executed contracts record **node-granular traces** (Beta posteriors, cost
  EWMAs, a precedence matrix) under the same keys the assembler scores by —
  bucketed per tenant, so your history personalizes only your picks.
- **`explore: true`** Thompson-samples picks from those posteriors: unproven
  alternatives get chances proportional to their remaining uncertainty, and
  exploration collapses onto winners as verified runs accumulate.
- **Learned orderings**: when your runs consistently finish one step before
  another, that order is stamped onto assembled contracts as learned edges —
  typed slots always outrank statistics, so a contradiction stays parallelism.

### Budgets that know you
- Three signals with three authorities: a **hard cap** refuses (402, never
  overridable), a **review threshold** holds until acknowledged (409), and a
  **behavioral comfort ceiling** learned from your own committed spending
  flags outliers with no configuration at all.
- Behavior is judged **per class of goal** (lavish gift spending never loosens
  the everyday budget, and vice versa) with **recency decay** (comfort tracks
  where your spending is *trending*; an old splurge stops justifying new ones).
- The linked wallet **never caps** — its balance may be a slice of your true
  assets, so it only ever asks for a review.

### Governance for irreversible actions
- A contract containing reserved actions is **held, not refused** — durable,
  tenant-scoped, decidable only by an authorized approver, with the run bound
  to the *original submitter* on approval. Holds expire on a TTL (swept and
  audited, never silently), and approvers subscribe to a **tenant-scoped SSE
  feed** of the hold lifecycle instead of polling.
- The desktop mirrors the same flow through its inbox, with bearer-token
  decisions verified server-side.

### Money, end to end
- **Settlement cycles** (`settle_all`): every noder, one period, one call —
  processor failures are contained per-noder and re-running the same period
  *is* the retry (paid noders replay receipts; nobody is ever paid twice).
- **Disputes claw back reserve-first**: an upheld dispute reverses exactly
  what the event minted, funds the shortfall from the noder's reserve, and
  leaves only honest debt that future earnings repay.
- **The reserve releases**: the holdback is scoped to the chargeback risk
  window — noders eventually receive **100% of undisputed earnings**.
- Fixed a real idempotency bug where a crashed operation poisoned its key
  forever (`IdempotencyLedger.release`).

### The desktop shell
- A dependency-free front-end served by the loopback at `GET /`: assemble
  (with budget verdicts and price breakdowns), tasks (clarifications, route
  confirmation, incidents, live timeline), the approval inbox, an **earnings
  screen** (balances, ledger, payout batches), **payout-account onboarding**
  with mirrored KYC status, skills search, and worker health. XSS-safe by
  construction (DOM building, no HTML templates), deep-linkable, light/dark.

## Verification

566 tests: unit and API suites, **property-style fuzzing of the money
invariants** (12 seeded economies × 50 random operations — reserve never
negative, money never minted, the books always match the processor),
**16-thread contention stress** on the shared stores, and **real-Chromium
end-to-end tours** of the shell. The setup scripts, packaging spec, CI
wiring, and UI routes are all pinned by tests so the onboarding story cannot
silently rot.

## Breaking changes

- `POST /v1/runs/contract` with reserved actions now returns **202
  `awaiting_approval`** (a durable hold) instead of 403; decide via
  `POST /v1/runs/contract/holds/{pending_id}`. The desktop confirm likewise
  holds instead of refusing.
- Spending-profile behavior changed by default: history is recency-weighted
  (`recency_decay: 0.9`; set `1.0` to restore flat history) and judged per
  goal class.
- The holds table is `pending_contracts` (was `desktop_pending_contracts`);
  holds made before upgrading should be re-confirmed.
- `compile_runnable` now returns a `CompiledContract` (blueprint + action
  ownership) rather than a bare `Blueprint`.

## New knobs

`earnings_ledger`, `payout_store`, `payout_adapter`, `noder_principal`
